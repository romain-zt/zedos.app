# Fast-track / mode urgent : règles produit v0

## Context

Le diagnostic produit (`docs/product/product-hics-diagnostic.md`) identifie **H1** : aucun parcours « c’est urgent, maintenant ». Le produit impose le tunnel clarify complet à tous les utilisateurs, y compris ceux sous deadline (pitch, DD, partenaire).

Six questions de cadrage ont été tranchées par le fondateur le 2026-06-03.

## Decision

| # | Question | Décision |
|---|----------|----------|
| 1 | Déclencheur urgent | **Tous les cas** : à la **création** du projet *et* **en cours** (« ce projet est devenu urgent » / re-bascule express) |
| 2 | Artefact urgent | **Livrable minimal** (structure réduite, sections essentielles) — pas un PRD complet avec placeholders |
| 3 | Clarify en urgent | **Minimum IA** : l’IA pose le **minimum** de questions nécessaires (pas cap fixe produit ; pas zéro systématique sauf import-only) |
| 4 | Crédits | **Même tarif** que le parcours standard (pas de réduction ni pack urgent dédié) |
| 5 | Transparence qualité | **Oui** : copy explicite **version express — à approfondir** sur la **page de partage** (et équivalent in-app owner) |
| 6 | Post-PRD en urgent | **Grisé** : surfaces feature-split, user-stories, delivery, etc. **visibles mais désactivées** avec message orienté objectif (pas masquées) |

### Décisions dérivées (alignement backlog)

| ID | Sujet | Tranché |
|----|-------|---------|
| D1 | Mode permanent vs one-shot | **Permanent** : bascule express ↔ standard à tout moment (cohérent avec Q1 « tous les cas ») |
| D4 | Import vs urgent | **Intentions distinctes** : « j’ai déjà un doc » ≠ « c’est urgent » ; peuvent se combiner (import + urgent) |
| D5 | Priorité backlog | **Fast-track avant import seul** si la douleur #1 est le temps (import reste P0 mais secondaire à H1) |

## Consequences / tradeoffs

**Pros :**
- Reconnaît le job-to-be-done « livrable maintenant » sans mentir sur la profondeur (copy express).
- Re-bascule urgent en cours couvre pivot deadline mid-projet.
- Même tarif crédits : simplicité ledger / pas de SKU urgent.
- Post-PRD grisé : l’utilisateur voit la suite possible sans frustration de tunnel caché.

**Cons / risques :**
- **Minimum IA** = variabilité (une session peut still feel long) — à monitorer vs cap UX optionnel plus tard.
- Livrable minimal ≠ PRD v2 complet : risque de confusion si copy express absente sur un canal (mitigé par Décision #5).
- Grisé post-PRD peut frustrer les cliqueurs — le message doit expliquer « approfondir d’abord » ou « passer en mode standard ».
