## P0 — blocks launch

### BUG-01 · Anonymous share links are completely broken

**Symptom:** an anonymous visitor opening a freshly minted, valid share link sees
"Not available — Something went wrong. Try again in a moment." The PRD never renders.
The product's read-only sharing journey (queue: `complete`) does not work at all.

**Repro:**
1. As owner: `POST /api/share/create` with a valid `prdVersionId` → 200, token returned.
2. In a clean browser: open `/share/<token>` → error card, no PRD.

**Root cause:** `apps/web/middleware.ts` → `isPublicPath()` whitelists `/share` (the page) but not
`/api/share/<token>` (the data). The client component `shared-prd-view.tsx` fetches
`/api/share/${token}`, middleware 307-redirects it to `/sign-in`, the JSON parse fails, the error
state renders.

```16:18:apps/web/middleware.ts
  if (pathname === '/api/stripe/webhook') return true;
  if (pathname.startsWith('/_next')) return true;
  if (pathname.startsWith('/share')) return true;
```

**Suggested fix (one line + tests):** treat `GET /api/share/<token>` and
`POST /api/share/<token>/access` as public in `isPublicPath`. Keep `/api/share/create` and
`/api/share/disable` protected (they already enforce `requireUser` at route level).
**Test gap that let this slip:** no Playwright spec opens a share link from an anonymous context —
add one (it's in QA_SYNOPSIS §9).

## P1 — fix before any marketing push

### BUG-02 · Clarify chat renders raw AI JSON to the user when the response fails validation

**Symptom:** the assistant bubble in the Clarifier shows raw JSON
(`...","confidence":"high","open_questions":[]}, {"id":"core_features",...`) instead of a question.
Progress score stays at 0%, no error message shown.

**Mechanism:** `app/api/projects/[id]/clarify/route.ts` streams the model output to the client
*before* validating it; the buffered result is schema-checked only server-side
(`persistAfterValidStream`), and on failure the server just logs
(`Clarify stream schema validation failed`) and returns. The client has already painted the raw
stream and gets no failure signal. Correctly, no credits are charged — but the user sees garbage.

Observed locally because the E2E stub returns a PRD-shaped payload for clarify; in production the
same path triggers whenever the model returns malformed/non-conforming output. This matches frozen
gap finding #19 (rule `74-contracts-zod.mdc` §11) and is likely a top source of the "really buggy"
perception.

**Suggested fix:** on buffered-parse failure, emit a terminal SSE error event; client replaces the
raw bubble with a retry-able error state. Never leave raw model text in the transcript.

### BUG-03 · Billing page throws 500s when Stripe is unreachable/misconfigured

**Symptom:** `/dashboard/billing` fires `GET /api/billing/invoices` and
`GET /api/billing/payment-method` → both 500 + toast "Impossible de charger la facturation".

**Mechanism:** both routes call the raw Stripe SDK with no try/catch and no Result wrapper
(`stripe.invoices.list(...)` at `app/api/billing/invoices/route.ts:38`). Any
`StripeAuthenticationError` / network error / deleted customer becomes an unhandled throw → 500.
Also a rules violation (raw vendor SDK in a 110-line route — `72-hexagonal-boundaries.mdc` §8.1).

**Suggested fix:** wrap Stripe calls → on failure return 502 with a typed error (or empty list +
`degraded: true`), and let the UI show "billing temporarily unavailable" instead of a generic error.

### BUG-04 · Dashboard first paint shows raw i18n keys

**Symptom:** on first dashboard load (reproduced at mobile viewport) the page renders translation
keys: `dashboard.hey E2E`, `dashboard.homeBaseDescription`, `dashboard.prdPathTitle`,
`credits.label`, `locale.fr`, `common.viewAll`, `dashboard.recentProjects` — then resolves.

**Mechanism:** the client i18n provider renders before messages are loaded and falls back to the
key string instead of holding a skeleton.

**Suggested fix:** block render behind the locale bundle (or SSR the messages); fallback must never
be the raw key.

## P2 — quality / coherence

### BUG-05 · Auth surfaces ignore the locale

Default locale is `fr` (middleware redirects `/` → `/fr`), but `/sign-in`, `/sign-up`, `/login`,
`/signup` and their error toasts ("Invalid email or password") are English-only. The credits page
mixes languages ("Prices shown are before tax. VAT or sales tax …" inside a French page). First
impression for a French user is an half-translated product.

### BUG-06 · Invalid or revoked share link: HTTP 200 + misleading copy

`/share/<bad-token>` returns **200** with "Something went wrong. Try again in a moment." An invalid
or revoked link is a permanent state, not a transient error — retrying won't help, and 200 invites
indexing/caching of a dead page. Expected per slice `read-only-sharing--revoke-link-and-noindex`:
a clear "link unavailable/revoked" state (ideally 404).

### BUG-07 · Sign-up redirect chain (4 hops) and duplicate auth routes

`/sign-up` → `/fr/sign-up` → `/signup` → `/fr/signup`. Two parallel route sets exist
(`/sign-in` + `/login`, `/sign-up` + `/signup`). Works, but slow first paint, noisy analytics,
and double maintenance surface. Pick one canonical pair and 308 the aliases directly.

### BUG-08 · v0 scope drift visible in the PRD tab

The PRD tab exposes **share password** ("Mot de passe de partage (optionnel, 8+ caractères)"),
**share expiry** ("Expiration (jours): 30") and **"Exporter en PDF"**. PRD v2 hard v0 exclusions
include advanced share controls (password, expiry). These shipped ungated on the main surface.
Either flag-gate them with the other Phase-1+ features or record a product decision (PD) that
re-scopes them in.

### BUG-09 · No public landing page

Unauthenticated `/` redirects straight to sign-in. There is no marketing surface in the app — a GTM
blocker (already discussed): ads/links need a landing page that demonstrates the product before
asking for an account.
